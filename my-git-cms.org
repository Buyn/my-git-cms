# 📄 requirements.md

# Requirements

## Overview

The system is a Git-based CMS with a frontend implemented in ClojureScript and a backend implemented in Python (FastAPI). The system prioritizes portability, simplicity, and future migration to a Clojure backend.

Content is primarily stored in a Git repository, while dynamic user-generated data is stored in a database.

---

## Core Principles

- Git is the single source of truth for content
- Frontend-first architecture
- Backend acts as a thin API layer
- Minimal coupling between backend and business logic
- Easy migration to alternative backend implementations

---

## Content Model

- All content is stored in a `/docs` directory
- Files are written in Org format
- Internal links use relative paths
- External links are allowed
- Media files are stored in a dedicated directory and referenced from content
- Content must support rendering of embedded media

---

## CMS Behavior

- Editing is performed through the frontend
- Changes are persisted via Git commits
- Versioning is handled by Git
- No traditional CMS database for content

---

## User Roles

### Anonymous User

- Can subscribe via email (no verification required)
- Can post comments on existing content

### Registered User

- Email verification required
- Can:

  * Manage profile
  * Edit own comments
  * React to comments

### Admin

- Full content management
- Write and edit articles
- Moderate comments and users

---

## Authentication

- Primary authentication via Git providers (GitHub/GitLab)
- Email-based identity for basic users
- Session-based or token-based access control

---

## Comments System

- Stored in database
- Linked to content via path or identifier
- Supports:

  * Editing (owner only)
  * Reactions (likes, etc.)
  * Moderation

---

## Hosting Constraints

- Must be deployable on PythonAnywhere
- No reliance on unsupported services (e.g., custom daemons, websockets-heavy features)
- Must work within WSGI/ASGI limitations of the platform

---

## Non-Goals

- Full WYSIWYG CMS
- Complex real-time features
- Tight coupling to any specific backend framework

---

## Future Considerations

- Migration to Clojure backend
- Possible headless CMS abstraction
- Static export capability

---

# 📄 design.md

# Design

## Architecture

Frontend (ClojureScript)

- SPA application
- Responsible for:

  * Rendering content
  * Editing interface
  * Client-side state management

Backend (FastAPI)

- Thin API layer
- Responsibilities:

  * Authentication
  * Git interaction
  * Comment storage and retrieval

---

## Directory Structure

```
/docs
  *.org
  /media
    images, videos, etc.
```

---

## Content Processing

1. Frontend loads `.org` files
2. Files are parsed into HTML (client-side or backend-assisted)
3. Links are resolved relative to `/docs`
4. Media paths are mapped to static serving endpoints

---

## Git Integration

- Backend performs:

  * Read operations (fetch content)
  * Write operations (commit changes)
- Each save action:

  * Creates commit
  * Uses authenticated user identity (Git provider if available)

---

## API Design

### Content

- GET /content/{path}
- POST /content/save

### Comments

- GET /comments/{content_id}
- POST /comments
- PUT /comments/{id}
- POST /comments/{id}/reaction

### Auth

- /auth/login (GitHub/GitLab)
- /auth/callback
- /auth/logout

---

## Database

Used only for:

- Comments
- Users (non-Git identities)
- Reactions

Suggested schema:

- users
- comments
- reactions

---

## Auth Flow

### Git आधारित users

1. OAuth via GitHub/GitLab
2. Backend receives token
3. User session created

### Email users

- Lightweight identity (no strict verification for anonymous)
- Optional verification for elevated permissions

---

## Frontend Editing

- Editor operates directly on Org files
- Changes tracked in memory
- Save triggers API call → commit

---

## Media Handling

- Stored in `/docs/media`
- Uploaded via API
- Referenced via relative paths

---

## Deployment (PythonAnywhere)

- Use ASGI-compatible setup
- Static files served via platform config
- Git operations via system git or library
- No background workers required

---

## Migration Strategy

To move away from FastAPI:

- Keep business logic outside API layer
- Avoid framework-specific dependencies
- Isolate Git and DB logic into services

---

## Tradeoffs

- No real-time updates (simplifies hosting)
- Manual CMS features instead of heavy frameworks
- Git complexity instead of DB simplicity

---

## Extensions (Optional)

- Static site export
- Search indexing
- Tagging system

---

# 📄 agent.md

# Agent Guide

## Project Overview

Git-based CMS with:

- Frontend: ClojureScript
- Backend: FastAPI
- Content: Org files in `/docs`
- Comments: Database

---

## Run Commands

Backend:

- uvicorn app.main:app --reload

Frontend:

- shadow-cljs watch app

---

## Common Tasks

- Start backend server
- Start frontend dev build
- Test API endpoints
- Validate Org parsing

---

## Conventions

- Keep backend thin
- No business logic inside routes
- Use services/modules for logic
- Prefer pure functions

---

## Git Rules

- Content changes = commits
- Do not bypass Git layer
- Always use relative paths

---

## Common Pitfalls

- Mixing business logic into FastAPI routes
- Hardcoding paths instead of using `/docs`
- Breaking relative links in Org files
- Forgetting PythonAnywhere limitations
- Overengineering auth

---

## Architecture Preferences

- Functional style where possible
- Clear separation:

  * API layer
  * Services
  * Data
- Avoid framework lock-in

---

## Frontend Notes

- Treat frontend as primary interface
- Editor is source of truth for content changes
- Avoid backend-driven rendering when possible

---

## Backend Notes

- FastAPI is replaceable
- Keep it stateless
- Avoid heavy dependencies

---

## Migration Awareness

- Code must be portable
- Do not depend on FastAPI-specific patterns
- Prepare for Clojure backend replacement

---

## Debug Strategy

- Check API boundaries first
- Validate Git operations
- Inspect data flow (frontend → API → storage)

---

## Philosophy

- Simplicity over features
- Control over convenience
- Replaceability over optimization
